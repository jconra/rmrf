MULTI-UNIT COMMANDER REFACTOR — verdict (2026-07-10, overnight run)
====================================================================

WHAT CHANGED
- AICommander refactored to SLOTS: per-unit state (nav cache, elevator ride, gate exit,
  recall trip, detour latches, stuck tracking, report card, debug snapshot) lives in slot
  records (freshSlot() in main.js); the commander binds a slot, runs the existing per-unit
  logic unchanged, unbinds. Team state (roster, scrap, doctrine, intel, kills, gambit) is
  shared by all slots.
- Unit cap = the team's ELEVATOR COUNT (min 1). RR.setUnitCap(n) forces it for testing.
- ?units=N (1-3): procedural maps build N-1 SATELLITE deploy pads outside each FOB ring
  (flanks first, rear fallback, water skipped); each lift fields one more simultaneous unit.
  Designed maps: place extra elevators in the editor and the cap follows automatically.
- Slot-aware external consumers: kill-credit loot, repair caps, gadget laying, ?nav overlay
  (one line per unit), spectate focus. tourney.cjs cmdOf/dbg updated (backward compatible).

KNOWN GAPS / FOR JACOB
- All of a team's units follow ONE shared doctrine (they converge on the same objective —
  a 3-unit push). Per-unit ROLE assignment is the next layer, and is exactly the commander
  decision the NN training is meant to learn.
- PvAI + ?units=3 = you alone vs three AI vehicles. Player-side wingmen don't exist yet.
- Designed maps: placeCampsFromConfig still assumes one elevator per team; extra placed
  elevators need a look before the editor advertises multi-lift.
- 3v3 A/B tournament: see below (appended when the run completes).

READ: multi-unit armies FINISH matches — +3 resolved, 73s faster, stalemates 5→2. This is
the first direct dent in the old "1 unit/team never sustains a siege" bottleneck (see
project_riposte_run_base_reduction): three units keep pressure on a base through individual
deaths. Per-unit stuck went DOWN; the new cost is idle-at-goal from all three units sharing
one doctrine objective (they arrive, then queue). Also seed 18 shows a resupply-congestion
pattern (multiple jotuns shuffling at the same depot for ~16s stretches) — a slot-aware
supply spread is a cheap future fix.

NEXT LAYER: per-slot ROLE assignment (siege/defend/raid instead of one shared objective) —
which is exactly the commander decision the NN training is planned to learn.

PER-SLOT DOCTRINE (ROLE DECK) — appended after Jacob's "per cmd doctrine" go-ahead
Each slot fields its OWN doctrine card from the persona's ROLE_DECK (slot 0 = the
archetype itself): warrior→[hunter,turtle], turtle→[turtle,warrior], hunter→[warrior,
turtle], rogue→[hunter,turtle]. Every 3-lift team keeps a home guard. Per-slot mission
fail-bans (_fail* moved into freshSlot). ?ailog overlay + log export show one block per
slot; death logs name the dead unit's own card.

SPREAD PADS + BARE FOB A/B — appended after Jacob's 3 playtest asks (2026-07-10)
Three changes shipped this pass: (1) side-elevator ACCENT bug fixed (all team lifts recolor,
not just the courtyard one); (2) satellite pads now SPREAD-SCAN (ring r=45..90, own-side,
≥34u apart) instead of hugging the FOB — "not all squished together"; (3) ?fobwalls=0 knob
drops the FOB ring/gates/towers, testing Jacob's theory that the elevator SHIELD now does the
anti-camp job the FOB towers used to.

READ:
- Spreading the pads cost a little: 30→28 resolved, +71s. Units field farther apart and
  spend more time regrouping/transiting to a shared objective (transit-stuck +52%). This is
  the price of the look Jacob asked for — a standoff seed (179: red alive 3, both bases at
  3/4 towers) is a genuine turtle draw, not a nav wedge.
- Dropping the FOB walls RECOVERS all of it and then some: same 28/30 resolution, but the
  FASTEST configuration measured (415s — beats even the squished 447s), idle-at-goal cut 38%
  (997 vs 1606), transit-stuck down 14%. Bases still defend with their own 4 corner towers;
  only the DEPLOY-POINT ring is gone. The elevator shield holds up as the anti-camp
  replacement — no camp-driven stalemate appeared. Two "stalemates" in each run are actually
  near-wins (one HQ reads dead=true; a classifier timing edge on mutual elimination), plus one
  real turtle standoff apiece.
- RECOMMENDATION: make BARE FOB the default for procedural multi-unit maps. It's faster,
  cleaner (less idle/stuck), holds resolution, and it removes ~half the repair-crew busywork
  (802→568 HP/match) that the FOB towers were generating. Keep the FOB-wall path behind
  ?fobwalls=1 for anyone who wants the fortified look. Net: the shield made the FOB fort
  redundant, exactly as Jacob guessed.
